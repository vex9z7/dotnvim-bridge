# Architecture and Directory Structure

Date: 2026-06-09

This document records the implementation structure for `dotnvim-bridge`, a Python MCP server that wraps pinned upstream `nvim-mcp==1.0.0` and exposes higher-level Neovim debug/config tools.

## Naming

Use one name across the repository, package metadata, and MCP command:

```text
repository:        dotnvim-bridge
distribution name: dotnvim-bridge
console command:   dotnvim-bridge
Python package:    dotnvim_bridge
```

The hyphenated name is used for the repository, Python distribution, and executable command. The underscored name is used for Python imports.

## Layout decision

Use a Python `src/` layout:

```text
dotnvim-bridge/
├── pyproject.toml
├── README.md
├── LICENSE
├── .env.example
├── .gitignore
├── uv.lock                         # generated by uv after dependency resolution
│
├── src/
│   └── dotnvim_bridge/
│       ├── __init__.py
│       ├── __main__.py             # supports: python -m dotnvim_bridge
│       ├── server.py               # MCP server entrypoint and tool registration
│       ├── settings.py             # NVIM_ADDRESS and response-limit configuration
│       ├── session.py              # adapter over pinned upstream nvim-mcp internals
│       ├── errors.py               # structured bridge/tool errors
│       ├── schemas.py              # stable response shapes / typed models
│       ├── limits.py               # line/byte truncation helpers
│       ├── lua_snippets.py         # reusable structured Neovim Lua snippets
│       │
│       └── tools/
│           ├── __init__.py
│           ├── debug_snapshot.py
│           ├── messages.py
│           ├── logs.py
│           ├── lsp.py
│           └── health.py
│
├── tests/
│   ├── __init__.py
│   ├── test_dependency_pin.py
│   ├── test_limits.py
│   ├── test_server_tools.py
│   ├── test_session.py
│   └── fixtures/
│       └── sample_responses.py
│
├── examples/
│   ├── codex-config.toml
│   └── codex-config-wrapper.toml
│
├── scripts/
│   ├── test-nvim-mcp-endpoint

│
├── docs/
│   ├── architecture.md
│   ├── setup.md
│   ├── roadmap.md
│   ├── core-requirements.md
│   ├── project-assessment.md
│   ├── debug-recipes.md
│   └── capability-boundary.md
│
└── openspec/
    └── changes/
        └── add-wrapper-nvim-mcp-with-pinned-upstream/
```

Rationale:

- `src/` layout catches packaging/import mistakes earlier than a flat layout.
- It matches the inspected upstream `paulburgess1357/nvim-mcp` package shape (`src/nvim_mcp/...`).
- It keeps implementation code separate from docs, examples, scripts, and OpenSpec material.
- It allows tests to exercise the installed/editable package rather than accidentally importing root-level files.


## MVP dependency strategy

For the MVP, maximize reuse of pinned upstream `nvim-mcp==1.0.0`. The project should avoid prematurely reimplementing Neovim msgpack-RPC, discovery, connection retry, state collection, diagnostics, command execution, or buffer reads when upstream already provides a working primitive.

The architectural intent is:

```text
thin stable communication layer
  -> mostly delegates to pinned upstream nvim-mcp
  -> owns only adaptation, error normalization, limits, and replacement seams

business/tool layer
  -> owns dotnvim-bridge behavior
  -> implements high-level debug/config workflows
  -> remains independent of upstream internals through session.py
```

This means `session.py` should be boring and small. Its job is not to become a second Neovim client implementation during MVP. Its job is to hide upstream details behind a local interface that can be replaced later.

Replacement of upstream internals is a later phase and should happen behind the same `session.py` adapter API so the public MCP tools and `tools/` modules do not need to change.

## Module responsibilities

### `server.py`

Owns the public MCP server entrypoint:

- creates the FastMCP server;
- registers high-level tools;
- exposes `main()` for the console script;
- should stay thin and avoid workflow-specific logic.

### `session.py`

Owns all direct coupling to upstream `nvim-mcp` internals:

- imports upstream manager/client classes;
- preserves the `NVIM_ADDRESS` connection contract;
- exposes a small adapter API for command execution, Lua execution, state, diagnostics, buffer reads, and bounded text retrieval;
- delegates to upstream `nvim-mcp` wherever practical during MVP instead of reimplementing transport primitives;
- is the main replacement seam if upstream internals become unsuitable.

No module under `tools/` should import upstream `nvim_mcp` internals directly.

### `tools/`

Each first-batch high-level MCP tool gets its own small module:

- `debug_snapshot.py` → `get_debug_snapshot`
- `messages.py` → `get_messages`
- `logs.py` → `get_logs_tail`
- `lsp.py` → `get_lsp_snapshot`
- `health.py` → `run_checkhealth`

Tool modules may call `session.py`, `lua_snippets.py`, `limits.py`, and `schemas.py`, but should not own transport/session setup.

### `lua_snippets.py`

Contains reusable Lua snippets sent ephemerally through Neovim RPC. These snippets are not installed into the user's Neovim config and are not persistent Neovim-side tools.

### `schemas.py`

Defines stable response shapes for public tool outputs. This protects agents from internal implementation churn and makes unit tests straightforward.

### `limits.py`

Centralizes response bounding:

- max lines;
- max bytes / characters;
- truncation metadata.

All text-heavy tools such as messages, logs, and checkhealth should use this module.

### `settings.py`

Centralizes runtime defaults and environment parsing:

- `NVIM_ADDRESS`;
- default message/log/health limits;
- optional debug flags.

## Test structure

Tests should start with mocked adapter/session responses and avoid requiring a live Neovim instance by default.

Required early tests:

- dependency pin test for `nvim-mcp==1.0.0`;
- limit/truncation behavior;
- tool response shape tests;
- adapter error normalization tests.

Live Neovim tests should be optional and gated by environment, for example:

```text
NVIM_ADDRESS=vex9z7.com:16667 RUN_LIVE_NVIM_TESTS=1 pytest
```

## Boundary rules

- Durable tool logic lives agent-side in `src/dotnvim_bridge/`.
- Neovim-side Lua is allowed only as ephemeral RPC snippets.
- MVP implementation should reuse pinned upstream `nvim-mcp` internals as much as practical, but direct imports must be isolated in `session.py`.
- Communication/session code should stay thin and stable; high-level behavior belongs in `tools/`.
- First-batch tools are read-oriented: they must not save files, quit Neovim, or invoke host shell commands.
